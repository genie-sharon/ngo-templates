# Contributing to Kindonar

Thank you for your interest in contributing to Kindonar. We welcome contributions from the community to help NGOs build better websites.

## Code of Conduct

By participating, you agree to maintain a respectful, inclusive, and harassment-free environment for everyone.

## Getting Started

1. Fork the repository
2. Clone your fork: `git clone https://github.com/your-username/kindonar.git`
3. Create a branch: `git checkout -b feature/your-feature-name`
4. Install dependencies: `npm install`
5. Start developing: `npm run dev`

## Development Workflow

1. Make your changes
2. Run type checking: `npm run typecheck`
3. Run linting: `npm run lint`
4. Run formatting: `npm run format:check`
5. Run tests: `npm test`
6. Commit with conventional commit message
7. Push and open a Pull Request

## What We Need Help With

### High Priority
- New section components for the builder
- Template improvements and new templates
- AI prompt engineering and refinement
- Accessibility testing and fixes
- Internationalization (i18n) support
- Performance optimization

### Medium Priority
- Documentation improvements
- Test coverage expansion
- Theme creation and design tokens
- Animation variant contributions
- UI/UX improvements to the builder

### Low Priority
- Code refactoring and cleanup
- Example and demo content
- Blog posts and case studies
- Translation of documentation

## Pull Request Guidelines

- Keep PRs focused on a single concern
- Include screenshots for UI changes
- Update relevant documentation
- Add or update tests
- Ensure CI passes all checks
- Reference any related issues

## Code Review Process

1. All PRs require at least one review
2. Maintainers may request changes
3. CI checks must pass before merging
4. Squash merge is preferred

## Questions?

Open a GitHub Discussion for questions, feature proposals, or general feedback. For bugs, open an Issue with a reproduction.
